# وينك — المستندات القانونية (Wenak Legal Documents)

هذا المجلد يحتوي على سياسة الخصوصية والشروط والأحكام الخاصة بتطبيق وينك، بالعربية والإنجليزية.

## الملفات

| الملف | المحتوى |
| --- | --- |
| `privacy-policy-ar.md` | سياسة الخصوصية — عربي (النسخة المرجعية) |
| `privacy-policy-en.md` | سياسة الخصوصية — إنجليزي |
| `terms-and-conditions-ar.md` | الشروط والأحكام — عربي (النسخة المرجعية) |
| `terms-and-conditions-en.md` | الشروط والأحكام — إنجليزي |

**ملاحظة مهمة:** النسخة العربية في كل مستند هي النص المرجعي المعتمد قانونًا. أي تعديل جوهري يجب أن يبدأ بالعربي أولًا، ثم يُعاد نقله للإنجليزي.

## ✅ قبل النشر — Placeholders لازم تتستبدل

كل مستند فيه رابط placeholder بيشاور على المستند التاني (السياسة بتشاور على الشروط، والعكس). لازم تستبدلهم بالروابط الفعلية بعد ما تجهّز الاستضافة:

- في `privacy-policy-ar.md` و `privacy-policy-en.md`: استبدل `رابط-الشروط-والأحكام-يُضاف-هنا` / `terms-and-conditions-link-to-be-added` برابط صفحة الشروط الفعلي.
- في `terms-and-conditions-ar.md` و `terms-and-conditions-en.md`: استبدل `رابط-سياسة-الخصوصية-يُضاف-هنا` / `privacy-policy-link-to-be-added` برابط صفحة السياسة الفعلي.

بعد ما تاخد رابط سياسة الخصوصية النهائي، هتحتاجه في مكانين خارج المستندات نفسها:
1. **Google Play Console** (قسم App content → Privacy Policy) وقت رفع التطبيق.
2. **متجر آبل** لاحقًا عند إطلاق نسخة iOS.

## هيكل الاستضافة المتفق عليه

- ريبو GitHub منفصلة عن ريبو التطبيق الأساسية، باسم `wenak-legal` (public، لأن GitHub Pages المجانية بتتطلب public).
- الهيكل المقترح:
```
/privacy-policy/
    index.html   → عربي (افتراضي)
    en.html      → إنجليزي
/terms/
    index.html   → عربي (افتراضي)
    en.html      → إنجليزي
README.md
```
- اللغة الافتراضية = **عربي**، مع رابط نصي واضح أعلى كل صفحة يودّي للنسخة التانية ("Read in English →" / "اقرأ بالعربي ←").
- مفيش JS-based auto language switcher — رابط نصي بسيط كافي وأوضح.
- محدش غيرك يقدر يعدّل على الريبو حتى لو public (القراءة عامة، الكتابة محصورة على الـ owner والـ collaborators فقط). يُفضّل تفعيل branch protection على main.

## استخدام نفس المحتوى داخل التطبيق (تبويب "المزيد")

نفس ملفات الـ Markdown دي ممكن تُستخدم مباشرة كـ asset جوّه Flutter (عبر مكتبة زي `flutter_markdown`) من غير ما تُكتب مرتين بصيغتين مختلفتين. أبسط طريقة:
- تُنسخ الملفات كـ assets داخل مجلد `assets/legal/` في مشروع Flutter.
- أو تُعمل عملية تحميل (download) من نفس رابط الاستضافة وقت فتح الصفحة، بحيث يكون المصدر واحد دايمًا ولا يحصل تعارض بين نسخة الموقع ونسخة التطبيق.

## نقاط لازم تترجع لها عند أي تغيير مستقبلي في المشروع

هذه المستندات مبنية على قرارات محددة اتفقنا عليها. أي تغيير في القرارات دي **لازم** يترافق بتحديث المستندين معًا (عربي وإنجليزي)، وتحديث تاريخ "آخر تحديث":

- **auto-purge بيانات الموقع (30 يوم):** السياسة مكتوبة على افتراض إن جدول `locations` بيتنضف تلقائيًا بعد 30 يوم. **لازم يتنفذ في الكود قبل النشر** (راجع كارد Trello "حذف تلقائي لتاريخ الموقع"). لو الرقم اتغيّر، حدّث القسم 5 في سياسة الخصوصية.
- **فيتشر تحميل/عرض سجل الموقع للمشتركين المميزين (30 يوم):** لو الفيتشر ده اتغيّر نطاقه (مثلاً زادت المدة، أو بقى متاح لغير الـ admin)، لازم يتحدّث القسم 4-أ في سياسة الخصوصية فورًا قبل تفعيل الفيتشر.
- **قرار سن الاستخدام (18+):** لو حصل تغيير معماري مستقبلي لدعم القُصّر رسميًا (البند المؤجل في Trello)، المستندين محتاجين إعادة كتابة كاملة تقريبًا (قسم السن في الاثنين + آلية موافقة أبوية)، مش مجرد تعديل بسيط.
- **الاشتراك عبر Apple IAP:** لما iOS ينزل، لازم يتحدّث القسم الخاص بالمدفوعات في الشروط ليشمل Apple بجانب Google Play.
- **أي مزوّد خدمة جديد** (SDK تحليلات، أداة دعم بالذكاء الاصطناعي، إلخ): لازم يُضاف لقائمة "مزوّدو الخدمة" في سياسة الخصوصية فور تفعيله، مش بعده.
- **تغيير النطاق (Domain):** لو `wenak.me` بقى جاهز، انقل الاستضافة من GitHub Pages المؤقتة لنطاق وينك الرسمي، وحدّث كل الروابط (Google Play Console، آبل، الـ asset جوّه التطبيق).
